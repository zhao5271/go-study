# 02 基础类型（转换、格式化、表达式）

> 这份笔记偏“知识库”：你未来写后台管理 API（分页、日志、错误码、RBAC 权限、参数校验）时，会不断回到这些基础能力。  
> 本笔记只聚焦：**基础类型的显式转换**、**fmt/strconv 的格式化与解析**、**运算符与表达式的工程坑**。

---

## 你需要先记住的 5 句“底层规则”

1) Go 的类型是“硬”的：不同数值类型之间不会自动转换（`int` ≠ `int64` ≠ `float64`）。  
2) **显式转换 ≠ 安全转换**：`int64→int` 可能溢出；`float64→int` 会截断小数。  
3) `string` 本质是只读字节序列；**`s[i]` 取到的是 byte**；`for range` 才是按 rune（Unicode）迭代。  
4) “数字转字符串”用 `strconv`（`Itoa/FormatInt`），不是 `string(n)`。  
5) 表达式里最常见的线上 bug：**整数除法截断**、**短路保护没写**、**优先级误读**、**位运算没括号**。

## 知识点 1：类型转换（Go 只允许显式转换）

### 一句话定义
Go 不做隐式数值类型转换；你要把一个值当成另一个类型使用，必须写显式转换：`T(x)`。

### 为什么重要（工程视角）
- **防止“看起来能跑，实际上埋雷”的隐式规则**：比如 JS/TS 里 `1 + "2"` 会变成 `"12"`；Go 把这种“隐式猜测”交给你显式表达。
- **在 API/DB 场景里到处都是跨类型边界**：query/body 是 string，DB 扫描常见 `int64`，业务逻辑常用 `int` 做下标/长度/分页。

### 关键点（要背下来的坑与取舍）
1) **不同整数类型不能直接混用**：`int` 和 `int64` 相加/比较都不行（编译期报错）。  
2) **`int` 的位数与平台相关**：32 位上 `int` 只有 32 位；64 位上是 64 位。大部分后端环境是 64 位，但你写库/工具函数时别假设。  
3) **浮点转整数会截断（朝 0）**：例如 `f := 1.9; int(f) == 1`，`f := -1.9; int(f) == -1`。如果你想“四舍五入”，要先用 `math.Round`。  
4) **byte vs rune**：
   - `byte` = `uint8`：更像“原始字节”（网络/文件/ASCII）。  
   - `rune` = `int32`：更像“Unicode 字符码点”（文字处理）。  
   - `string` 的“索引”是 byte；“range”才是 rune。
5) **无类型常量（untyped constant）有点“像 TS 的字面量”**：
   - `const x = 1` 本身没有固定类型，能按上下文赋给 `int/int64/float64`（只要能表示）。  
   - `var x = 1`（变量）一旦推导出类型（通常是 `int`），后面就必须显式转换才能参与其他类型运算。

### 基础类型速查（最常用的一小撮）
- `bool`：条件判断；零值 `false`  
- `string`：UTF-8 字节序列（只读）；零值 `""`  
- `int`：最常用整数（长度/下标/分页）；位宽随平台；零值 `0`  
- `int64`：跨边界更稳定（DB BIGINT、时间戳）；零值 `0`  
- `float64`：小数/比例（注意精度与舍入）；零值 `0`  
- `byte`：原始字节；常见于 `[]byte`  
- `rune`：Unicode 码点；常见于 `for range s` 的遍历变量

### 业务场景落地（后台管理 API）
- **分页**：`page,size` 从 query 来（string）→ 转 int → 校验边界 → 计算 `offset=(page-1)*size`。  
- **ID**：DB 主键常用 `BIGINT`（`int64`），服务/协议层如果用 `int`，就要明确“是否允许溢出”与“如何处理非法/过大 ID”。  
- **RBAC**：权限常用 bit flag（位标志），位运算会强依赖整数类型与位宽概念（建议用 `uint64`/自定义类型）。

### 知识点运用示例

#### 示例 1.1：`int` + `int64` 如何正确相加（以及为什么要这样）
主要代码片段：
```go
x := 42
var big int64 = 1
sum := int64(x) + big
fmt.Printf("sum=%d type=%T\n", sum, sum) // Output: sum=43 type=int64
```

你会在 DB 扫描出来的 `int64(id)` 和应用里常见的 `int(page)` 混用时不断碰到这个问题：**要么统一全链路类型，要么在边界处转换并校验**。

#### 示例 1.2：`int64 → int` 的“安全转换”模板（避免溢出）
主要代码片段：
```go
func int64ToInt(v int64) (int, error) {
	if strconv.IntSize == 32 {
		if v < math.MinInt32 || v > math.MaxInt32 {
			return 0, fmt.Errorf("int overflow: %d", v)
		}
	}
	return int(v), nil
}
```
说明：
- `strconv.IntSize` 是编译目标平台的 `int` 位数。  
- 64 位平台上通常不用这段判断，但**写成可复用模板能避免迁移/交叉编译时踩坑**。

#### 示例 1.3：`string` 的索引是 byte，`range` 才是 rune
主要代码片段：
```go
s := "中A"
fmt.Printf("len=%d\n", len(s)) // Output: len=4（UTF-8 编码：中占 3 字节，A 占 1 字节）

fmt.Printf("s[0]=%d\n", s[0]) // Output: s[0]=228（说明：这是 UTF-8 第 1 个字节，和字符本身不是一回事）

for i, r := range s {
	fmt.Printf("i=%d r=%q\n", i, r) // Output: i=0 r='中'  i=3 r='A'
}
```
工程结论：
- “按字符数截断/计数/校验”要用 `range`（或 `utf8.RuneCountInString`），不能用 `len(s)`。  
- “按字节协议/网络/文件”可以用 `len` 和 `[]byte`。

## 知识点 2：格式化（fmt）与字符串转换（strconv）

### 一句话定义
`fmt` 负责“把值按某种格式表达出来”（打印/拼字符串）；`strconv` 负责“字符串 ↔ 基础类型”的**稳定可控**转换（解析/格式化）。

### 为什么重要（工程视角）
- 你要写日志、错误信息、调试输出：**格式不清楚就会误判**（尤其是 string/[]byte/rune 的差异）。  
- 你要解析 query/body 的 string 参数：**解析失败必须能给出可追踪的错误信息**，并且要能对边界做校验（size 1..100）。

### 关键点（要背下来的坑与取舍）
1) **`string(n)` 不是数字转字符串**：`string(65) == "A"`（把整数当 Unicode 码点/字节转字符）。  
2) **解析用 `strconv`，展示用 `fmt`**：
   - `strconv.Atoi("123")` → `int`  
   - `strconv.ParseInt("123", 10, 64)` → `int64`（可控 base/位宽）  
   - `fmt.Sprintf("id=%d", id)` → 拼可读字符串
3) `fmt` 常用动词（足够覆盖 90% 后端调试/日志需求）：
   - `%v` 默认；`%+v` 带字段名（struct）；`%#v` Go-syntax（定位结构特别快）  
   - `%T` 类型；`%q` 带引号字符串/字符；`%t` bool  
   - `%d` 十进制整数；`%x` 十六进制（常用于 byte/ID/trace）

### fmt 常用动词速查表（建议收藏）
| 目标 | 推荐写法 | 备注 |
|---|---|---|
| 打印值（默认） | `%v` | 最常用 |
| 打印 struct（带字段名） | `%+v` | 排障很快 |
| 打印 Go 语法表示 | `%#v` | 适合定位结构/默认值 |
| 打印类型 | `%T` | 判断“我以为的类型”和“真实类型” |
| 打印字符串（带引号） | `%q` | 能看清空格/转义 |
| 打印整数 | `%d` | 十进制 |
| 打印浮点（指定小数位） | `%.2f` | 金额不要直接用 float（后续会讲） |
| 打印十六进制 | `%x` | `[]byte` / hash / trace 常用 |

### 业务场景落地（后台管理 API）
- **参数解析**：`pageStr := r.URL.Query().Get("page")` → `Atoi` → 校验 → 计算 offset。  
- **错误上下文**：`fmt.Errorf("list users page=%d size=%d: %w", page, size, err)`（把关键输入带上，排障速度翻倍）。  
- **审计日志**：常见写法 `fmt.Sprintf("user_id=%d action=%s", uid, action)`；二进制字段用 `%x` 打印更直观。

### 知识点运用示例

#### 示例 2.1：数字与字符串互转（以及 `string(n)` 的坑）
主要代码片段：
```go
n := 65
fmt.Printf("n=%d\n", n)                 // Output: n=65
fmt.Printf("string(n)=%q\n", string(n)) // Output: string(n)="A"
fmt.Printf("Itoa(n)=%q\n", strconv.Itoa(n)) // Output: Itoa(n)="65"
```

#### 示例 2.2：为什么我更推荐 `ParseInt`（而不是只会 Atoi）
主要代码片段：
```go
v64, err := strconv.ParseInt("9223372036854775807", 10, 64)
if err != nil {
	fmt.Printf("ParseInt err=%v\n", err) // Output: （输出可能变化/不固定：错误信息随输入/实现变化）
	return
}
fmt.Printf("v64=%d type=%T\n", v64, v64) // Output: v64=9223372036854775807 type=int64
```
工程取舍：
- `Atoi` 只能得到 `int`，位宽随平台；你在 DB/协议层需要稳定 `int64` 时，`ParseInt(..., 64)` 更可控。  
- 解析 HTTP 参数时，`ParseInt` 还允许你显式限制位宽（比如只允许 32 位范围）。

#### 示例 2.4：解析 query 参数的“最小通用策略”（默认值 + 边界 + 错误上下文）
主要代码片段：
```go
func parseIntQuery(v string, def int, min int, max int) (int, error) {
	if v == "" {
		return def, nil
	}
	n, err := strconv.Atoi(v)
	if err != nil {
		return 0, fmt.Errorf("must be int: %w", err)
	}
	if n < min || n > max {
		return 0, fmt.Errorf("out of range [%d..%d]: %d", min, max, n)
	}
	return n, nil
}
```
工程提示：
- 这类函数非常适合抽到“参数解析工具包”（后续抽包时会落地），避免每个 handler 各写一套边界规则。

#### 示例 2.3：`%q`、`%T`、`%x` 在调试时非常值钱
主要代码片段：
```go
raw := []byte{0xDE, 0xAD, 0xBE, 0xEF}
fmt.Printf("raw=%x\n", raw) // Output: raw=deadbeef

s := "hello"
fmt.Printf("s=%q type=%T\n", s, s) // Output: s="hello" type=string
```

## 知识点 3：运算符与表达式（整数除法、短路、优先级、位运算）

### 一句话定义
表达式就是“按规则算出一个值”；Go 的规则很简单，但**整数除法**、**短路**、**优先级**、**位运算**最容易让业务代码出 bug。

### 为什么重要（工程视角）
- **分页/统计**：整数除法截断导致页数/比例算错（线上数据看着“差一点点”，排查很费劲）。  
- **健壮性**：短路是你避免 nil dereference 的第一道护栏。  
- **RBAC**：位运算是权限/状态位的高性价比表达（存储/查询方便），但写错括号就会错得很隐蔽。

### 关键点（要背下来的坑与取舍）
1) **整数除法必截断**：`5/2 == 2`；如果要小数，至少一边转 `float64`。  
2) **短路保护要写在左边**：`p != nil && *p > 0` 是安全的；反过来就会 panic。  
3) **优先级别靠猜，读代码的人会痛苦**：工程上建议“关键逻辑都加括号”。  
4) **位运算一定要写成 `(mask & flag) != 0`**：不要省括号（可读性/避免误读）。
5) **可比较性（comparable）**：基础类型大多可 `==`，但 `slice/map/func` 不能用 `==` 比较（除了和 `nil` 比）。

### 业务场景落地（后台管理 API）
- **分页 offset**：`offset := (page-1)*size` + 明确边界（page>=1；size 1..100）。  
- **过滤条件**：`if search != "" && len(search) >= 2 { ... }`（短路保护：空字符串就不再做后续计算）。  
- **权限**：用 bit flag 表示多个权限：READ/WRITE/EXPORT；DB 存 `BIGINT UNSIGNED`；查询时可做位运算或服务层判断。

### 知识点运用示例

#### 示例 3.1：整数除法截断（以及如何得到小数）
主要代码片段：
```go
fmt.Printf("5/2=%d\n", 5/2) // Output: 5/2=2
fmt.Printf("5/2=%.1f\n", float64(5)/2) // Output: 5/2=2.5
```

#### 示例 3.2：短路保护避免 nil 指针崩溃
主要代码片段：
```go
var p *int
if p != nil && *p > 0 {
	fmt.Println("p>0") // Output: （不会打印，因为 p==nil，短路）
} else {
	fmt.Println("short-circuit ok") // Output: short-circuit ok
}
```

#### 示例 3.3：分页 offset 的最小可复用函数（边界写死，行为稳定）
主要代码片段：
```go
func calcOffset(page, size int) (int, error) {
	if page < 1 {
		return 0, fmt.Errorf("page must be >= 1")
	}
	if size < 1 || size > 100 {
		return 0, fmt.Errorf("size must be 1..100")
	}
	return (page - 1) * size, nil
}
```

#### 示例 3.4：位标志（bit flag）表达 RBAC 权限
主要代码片段：
```go
type Perm uint64

const (
	PermRead Perm = 1 << iota
	PermWrite
	PermExport
)

func hasPerm(mask, p Perm) bool { return (mask&p) != 0 }

mask := PermRead | PermExport
fmt.Printf("mask=%b\n", mask) // Output: mask=101
fmt.Printf("hasRead=%t\n", hasPerm(mask, PermRead)) // Output: hasRead=true
fmt.Printf("hasWrite=%t\n", hasPerm(mask, PermWrite)) // Output: hasWrite=false
```
工程提醒：
- 位标志非常适合“一个用户拥有多个权限”的表达；但当权限要动态扩展/需要审计/需要解释性很强时，也可以选择“用户-角色-权限”关系表（本项目后续会讲 RBAC 的关系建模）。
