Day 02：函数与错误处理

TS/Node → Go：从 try/catch 思维切到 (value, error) 工程习惯

核心命题

把错误当作“值”来设计与传递：用 wrap 加上下文，用 errors.Is/As 做稳定匹配，用 defer 管资源，用测试把边界锁死。

深层原因

错误是值，而不是异常

Go 倾向显式 `(value, error)`，避免 try/catch 漏捕或隐藏控制流；每一层都能加上下文并保持可匹配语义。

wrap 的取舍：语义 vs 抽象边界

用 `%w` 会把底层错误语义暴露给调用方（errors.Is/As 可匹配），等于把它变成 API 承诺；不想承诺就用 `%v` 只给人看。

defer 让资源释放不再靠记忆力

打开资源后立刻 defer 关闭，避免多 return 分支遗漏；并牢记三条规则：参数求值时机、LIFO、可修改具名返回值。

测试是第一等公民

table-driven tests + t.Run 子测试，把边界情况（not found、invalid input 等）固化为可重复验证的工程资产。

if init 与 := shadowing（必踩坑）

目标：理解为什么推荐 if init，同时避免 err 被 shadowing 导致的“外层 err 没更新”。

go run ./cmd/day02/03_shadowing

// pitfall: := creates a new err (outer err unchanged)
if _, err := users.FindUserSentinel(2); err != nil {
	fmt.Printf("inside if: err=%v\n", err) // Output: inside if: err=find user 2: user not found
}
fmt.Printf("after if: outer err=%v\n", err) // Output: after if: outer err=<nil>
2.4

defer：LIFO / 参数求值 / 具名返回值

目标：记住三条规则，并会用 defer 做资源释放与返回值修饰。

go run ./cmd/day02/04_defer

代码：go-learning/cmd/day02/04_defer/main.go

// defer args are evaluated immediately
x := 1
defer fmt.Printf("defer x=%d\n", x) // Output: defer x=1
x = 2
fmt.Printf("now x=%d\n", x) // Output: now x=2
2.5

panic vs error + recover（边界兜底）

目标：业务失败返回 error；panic 仅用于不可恢复错误；recover 只能在 defer 中兜底。

go run ./cmd/day02/05_panic_recover

代码：go-learning/cmd/day02/05_panic_recover/main.go

func safe(fn func()) (recovered any) {
	defer func() {
		recovered = recover()
	}()
	fn()
	return nil
}
2.6

table-driven tests（用测试锁边界）

目标：掌握 tests 表 + t.Run 子测试的结构化写法。

go test ./... -run TestFindUser

测试文件：go-learning/internal/day02/users/users_test.go

tests := []struct {
	name string
	id   int
	wantErrIs error
}{
	{"found", 1, nil},
	{"not found", 2, ErrUserNotFound},
}

for _, tt := range tests {
	t.Run(tt.name, func(t *testing.T) {
		_, err := FindUserSentinel(tt.id)
		if tt.wantErrIs != nil && !errors.Is(err, tt.wantErrIs) {
			t.Fatalf("... ")
		}
	})
}

深远影响

  • 1
    对外错误语义要稳定： 对外暴露 sentinel/type 语义；不要把底层库错误泄漏成 API 承诺（除非你愿意长期维护）。
  • 2
    上下文要短： wrap 时避免“failed to ...”层层堆叠，保持链条可读。
  • 3
    入口统一转换错误： 在 handler/入口层把 error 转成 HTTP 响应；内部只返回 error，不直接写响应。

多元视角

TS/Node 对照：从 try/catch 到显式错误流

"Go 不让错误偷偷穿过边界：每个调用点必须面对 err，这会让大型服务更稳定、更可测试。"

⚠ 易错点纠偏

误区：用 err == ErrXxx 判断就够了

事实：一旦你 wrap 了错误，必须用 errors.Is/As 才能稳定匹配。

误区：看到 error 就 %w 一把梭

事实：wrap 会暴露底层错误语义，可能破坏抽象边界；不想承诺就用 %v。

误区：panic 就是 Go 的 throw

事实：业务失败返回 error；panic 只用于不可恢复的程序错误，recover 仅做边界兜底。
