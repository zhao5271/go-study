Go 语法基础：变量与零值（Lesson 01.1）

面向 Vue3 / TypeScript / Node 工程师的 Go 全栈入门第一步

核心命题

把 Go 的“零值 + 静态强类型 + 显式转换/显式错误”思维切过来，后面写 API（HTTP + MySQL）才稳、才可维护。

深层原因

零值（Zero Value）取代 undefined

Go 变量声明后立即可用：数值=0、字符串=""、布尔=false、引用类型=nil。用确定性降低“没初始化就用”的工程风险（对比 TS/JS 常见 undefined/null 坑）。

短变量声明 := 提升局部可读性

只在函数内使用，语法更短、类型推导更顺手；但依然是静态强类型，不是 JS 运行期随意变类型。

不做隐式类型转换（更早暴露问题）

int 不能直接加 int64，需要显式转换。用“多写一点转换”换取更可控的精度/溢出边界，把问题前置到编译期。

编译期严格约束（未使用即报错）

未使用的变量/导入会直接编译失败。取舍是更严格，但换来更干净的依赖与更少的“死代码/脏 import”。

发展历程

1.1

var 声明 + 零值（Zero Value）

目标：建立“声明即初始化”的心智模型，告别 undefined。

package main

import "fmt"

func main() {
    fmt.Println("== 01.1A var + zero values ==") // Output: == 01.1A var + zero values ==

    var count int
    var name string
    var ok bool
    fmt.Printf("count=%d name=%q ok=%v\n", count, name, ok) // Output: count=0 name="" ok=false
}

运行：cd /Users/zhang/Desktop/go-study/codex/go-learning 然后 go run ./cmd/lesson01_01a_var_zero

1.2

:= 短变量声明（函数内）

目标：像 TS 的类型推导一样舒服，但保持 Go 的静态强类型。

package main

import "fmt"

func main() {
    fmt.Println("== 01.1B := short declaration ==") // Output: == 01.1B := short declaration ==

    x := 42
    fmt.Printf("x=%d type=%T\n", x, x) // Output: x=42 type=int
}

运行：go run ./cmd/lesson01_01b_short_decl

1.3

显式类型转换（int vs int64）

目标：适应 Go 的“无隐式转换”，把边界问题前置到编译期。

package main

import "fmt"

func main() {
    fmt.Println("== 01.1C explicit conversion (int vs int64) ==") // Output: == 01.1C explicit conversion (int vs int64) ==

    x := 42
    var big int64 = 1
    sum := int64(x) + big
    fmt.Printf("int64(x)+big=%d type=%T\n", sum, sum) // Output: int64(x)+big=43 type=int64
}

运行：go run ./cmd/lesson01_01c_conversion

1.4

const 常量

目标：理解 Go 的 const 更偏“编译期常量”，而不仅是“不可重新赋值”。

package main

import "fmt"

func main() {
    fmt.Println("== 01.1D constants ==") // Output: == 01.1D constants ==

    const pi = 3.14159
    fmt.Printf("pi=%.2f type=%T\n", pi, pi) // Output: pi=3.14 type=float64
}

运行：go run ./cmd/lesson01_01d_constants

深远影响

  • 1
    局部默认用 :=： 更短、更清晰；只有在包级变量、需要零值、或要指定类型宽度时用 var。
  • 2
    类型宽度重要时显式声明： 时间戳、字节大小、DB 字段等建议用 int64/uint64 等明确类型，避免跨平台差异。
  • 3
    养成 gofmt / go run / go test 的循环： Go 的工程体验依赖工具链：先格式化再运行/测试，后续做 API/MySQL 会更顺。

多元视角

从 TS/Node 视角看零值

"Go 把“可能不存在/没初始化”的问题用零值语义收敛，减少运行期 undefined/null 分支爆炸。"

从 TS/Node 视角看类型转换

"Go 宁愿让你写显式转换，也不允许类型在不知不觉中变了；这对大型服务的可维护性很关键。"

⚠ 易错点纠偏

误区：“:= 到处都能用”

事实：:= 只能在函数内使用；包级变量只能用 var。

误区：“int 和 int64 反正都是整数，直接加就行”

事实：Go 不允许隐式转换，必须显式把其中一个转换成另一个类型。

误区：“声明了变量/导入了包不使用也没关系”

事实：未使用的变量/导入会直接编译失败（这是一种工程约束）。
